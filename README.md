## 中文文档

### 项目简介

Chronicle 是一个以 RPG / 编年史风格呈现的个人主页与博客，使用纯静态前端（HTML + CSS + JavaScript）+ JSON 数据驱动。整套站点不依赖任何后端框架或静态站点生成器，可以直接放在任何静态托管平台上运行。

### 技术栈

- 纯静态 HTML5 / CSS3 / Vanilla JavaScript（单 IIFE，无构建步骤）
- [marked.js](https://github.com/markedjs/marked) 用于随笔 Markdown 渲染
- Cloudflare Worker 作为订阅接口代理，避免前端泄漏 GitHub Token
- GitHub Actions 用于：自动更新 `lastUpdate`、订阅者邮件通知、自动重建 `sitemap.xml` / `feed.xml`
- PWA：Web App Manifest + Service Worker，支持离线访问与缓存策略

### 目录结构

```text
.
├─ index.html                # 页面入口
├─ manifest.json             # PWA 应用清单
├─ sw.js                     # Service Worker（缓存与离线）
├─ favicon.svg               # 站点图标
├─ icon-192.png              # PWA 图标 192×192
├─ icon-512.png              # PWA 图标 512×512
├─ 404.html                  # 404 页面
├─ robots.txt
├─ sitemap.xml               # 由 Action 自动生成
├─ feed.xml                  # 由 Action 自动生成
├─ CNAME                     # GitHub Pages 自定义域 (miracleyang.com)
├─ .nojekyll                 # 关闭 GitHub Pages 默认构建流程
├─ style/
│  ├─ chronicle.css
│  └─ chronicle.js
├─ database/
│  ├─ profile.json
│  ├─ vocation.json
│  ├─ being.json
│  └─ documents/
└─ .github/
   ├─ ISSUE_TEMPLATE/
   │  └─ subscribe.yml             # 订阅 Issue 表单模板
   └─ workflows/
      ├─ auto-update-date.yml      # lastUpdate 自动更新
      ├─ notify-subscribers.yml    # 订阅邮件群发
      ├─ gen-sitemap-feed.yml      # sitemap / feed 自动生成
      └─ notify-template.txt
```

### 部署

本仓库直接面向 GitHub Pages 部署：

1. 将代码推送到 `main` 分支。
2. 仓库 Settings → Pages → Source 选择 `Deploy from a branch` → `main` / `(root)`。
3. 根目录的 `CNAME` 已指向自定义域 `miracleyang.com`，在 DNS 处配置 A / CNAME 记录后启用 Enforce HTTPS。
4. `.nojekyll` 文件用于跳过默认构建流程并允许下划线开头的文件被服务。

### 订阅功能

- 前端将邮箱写入仓库 Issues（标题 `[subscribe] <email>`），通过 Cloudflare Worker 代理鉴权调用 GitHub API。
- 通过手动触发 `.github/workflows/notify-subscribers.yml` 群发更新邮件，邮件正文使用响应式 HTML 模板。
- 仓库提供 Issue 表单模板（`.github/ISSUE_TEMPLATE/subscribe.yml`），订阅者也可以在 GitHub Issues 页面直接订阅。
- 工作流会用严格的邮箱正则校验来源，并在失败时自动创建 `[CI Failure]` Issue。

### Service Worker 版本更新

`sw.js` 顶部维护一个手动版本常量：

```js
const CACHE_VERSION = '20260613-1';
```

发布新版本时请按 `YYYYMMDD-N` 的格式手动递增（同日多次发版只递增尾号）。新版本激活后会自动清理所有不匹配该前缀的旧缓存。

## English Documentation

### Overview

Chronicle is a personal homepage and essay site presented in an RPG / chronicle style. It is a fully static front end (HTML + CSS + JavaScript) driven by JSON content, with no backend framework and no static-site generator.

### Tech Stack

- Plain HTML5 / CSS3 / vanilla JavaScript (single IIFE, no build step)
- [marked.js](https://github.com/markedjs/marked) for Markdown rendering in essays
- A Cloudflare Worker proxy fronts the GitHub API so no PAT is ever exposed client-side
- GitHub Actions handle: `lastUpdate` auto-bumping, subscriber email blasts, and sitemap/feed regeneration
- PWA: Web App Manifest plus a Service Worker for offline-friendly caching

### Deployment

1. Push to the `main` branch.
2. Repo Settings → Pages → Source: `Deploy from a branch` → `main` / `(root)`.
3. The `CNAME` file points the site at `miracleyang.com`. Configure A / CNAME records at your DNS provider and enable Enforce HTTPS.
4. The `.nojekyll` file disables the default build pipeline and allows files starting with `_` to be served.

### Subscriptions

- The front end records subscribers as GitHub Issues titled `[subscribe] <email>`, going through a Cloudflare Worker so the GitHub PAT stays server-side.
- `notify-subscribers.yml` can be triggered manually to blast an email update; the body uses a responsive HTML template.
- An Issue Form template (`.github/ISSUE_TEMPLATE/subscribe.yml`) lets users subscribe directly from GitHub.
- The workflow runs a strict email regex and opens a `[CI Failure]` Issue if anything goes wrong.

### Service Worker Versioning

`sw.js` exposes a manual version constant at the top:

```js
const CACHE_VERSION = '20260613-1';
```

Bump it manually on every release using `YYYYMMDD-N` (increment the suffix when shipping multiple times in one day). Activation purges any cache whose key does not start with the new version.
